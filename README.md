# Policy Number Challenge (Node.js)

This JavaScript project implements OCR parsing and policy-number validation/correction as a Node.js CLI.
It is organized around four challenge user stories (`us1` to `us4`) and supports any user-provided input/output file paths.

## Requirements

- Node.js 20+

## Quick Start

```bash
npm install
npm test
```

## Terminology

- `story` - Functional mode selected by `--story` in CLI.
- `us1` - OCR parsing only; prints decoded 9-digit numbers.
- `us2` - Checksum validation; appends `ERR` for invalid numbers.
- `us3` - Findings report; appends `ILL` (unreadable) or `ERR` (checksum failure).
- `us4` - Correction report; attempts one-segment repair and emits corrected numbers or `AMB`/`ILL`.
- `fixture` - Sample input/output file used by tests.
- `expected_output` - Golden reference output committed in repo.
- `generated_output` - Output generated by tool flow for parity checks in tests.

## Status Codes

- `ERR` - Number is readable but fails checksum.
- `ILL` - Number contains unreadable digit(s), for example `?`.
- `AMB` - Multiple valid corrections exist (ambiguous repair).

## Assumptions and Constraints

- Input files follow the OCR challenge format: 3 text rows per account number plus a separator row.
- Each account number is expected to be 9 digits wide in OCR layout.
- CLI `--story` defaults to `us1` when omitted.
- Exit code `2` is used for CLI/input/output errors (invalid args, unsupported story, unreadable input, unwritable output).
- US4 correction scope is one-segment change per candidate, following challenge rules implemented in `src/ocr/us4/repair.js`.
- Runtime targets Node.js 20+.

## How Input and Output Work

- Input is always an OCR text file path (`<input-path>`).
- Output can be:
  - printed to stdout (default), or
  - written to a file via `--out <output-path>`.
- This behavior is identical for `us1`, `us2`, `us3`, and `us4`.

## CLI Usage

```bash
node ./bin/ocr.js <input-path> [--story us1|us2|us3|us4] [--out <output-path>]
```

## Project Structure

- `bin/ocr.js` - CLI entrypoint.
- `src/ocr/parser.js` - OCR row/chunk parser and digit decoding flow.
- `src/ocr/us2/` - checksum validation logic.
- `src/ocr/us3/` - findings reporting logic.
- `src/ocr/us4/` - correction and ambiguity handling logic.
- `src/ocr/reporting/` - reporting-stage strategy/factory orchestration.
- `test/ocr.test.js` - comprehensive tests across all stories and error paths.
- `test/fixtures/` - canonical fixtures by story (`us1`..`us4`).

## Technical LLD

### Architectural Style

- Layered CLI architecture with story-based reporting strategies.
- Parsing, validation, reporting, and repair are separated by responsibility.

### Design Patterns Used

- `Strategy Pattern`
  - Story behavior is encapsulated per mode in `src/ocr/reporting/strategies.js`.
  - Strategies: `Us1ReportingStrategy`, `Us2ReportingStrategy`, `Us3ReportingStrategy`, `Us4ReportingStrategy`.
- `Factory Pattern`
  - `createReportingStage` in `src/ocr/reporting/reportingStageFactory.js` resolves `--story` to the correct strategy.
- `Composition`
  - `Us4ReportingStrategy` composes `US4FindingsWriter`, which composes `CorrectionResolver`.

### Core Components

- `src/ocr/parser.js`
  - `PolicyFileParser` reads OCR blocks and decodes digit rows into policy numbers.
  - `DigitDecoder` converts OCR chunks to canonical digits.
- `src/ocr/us2/validator.js`
  - `ChecksumValidator` applies the weighted checksum rule.
- `src/ocr/us3/report.js`
  - `FindingsWriter` emits `ILL`/`ERR` findings.
- `src/ocr/us4/repair.js`
  - `CorrectionResolver` performs one-segment candidate generation and disambiguation.
  - `CorrectionResult` standardizes output line formatting.
- `src/ocr/cli.js`
  - Parses CLI args, selects reporting stage, and routes output to stdout or `--out` file.

### Runtime Flow

1. CLI reads `<input-path>` and optional `--story`, `--out`.
2. `PolicyFileParser` parses OCR text into decoded entries.
3. `createReportingStage(story)` selects strategy for `us1`..`us4`.
4. Strategy builds story-specific output lines.
5. Lines are printed to stdout or written to the user-provided output path.

### Error Handling

- Invalid CLI shape (missing input, extra args, unsupported story) returns code `2`.
- Input read errors and output write errors return code `2` with a descriptive message.

### Extensibility Notes

- To add `usN`, implement a new strategy in `src/ocr/reporting/strategies.js`.
- Register it in `src/ocr/reporting/reportingStageFactory.js`.
- Add fixtures and tests to keep contract and coverage consistent.

## Docker

This repository uses a single multi-stage `Dockerfile` with two stages: `base` and `runtime`.

### 1) Build base stage

```bash
docker build --target base -t policy-ocr-base:latest .
```

### 2) Build runtime stage

```bash
docker build --target runtime -t policy-ocr:latest .
```

Run with user-provided paths (mounted into container):

```bash
docker run --rm -v "${PWD}:/work" policy-ocr:latest --story us1 /work/test/fixtures/us1/us1_input_sample1.txt --out /work/tmp_us1_out.txt
docker run --rm -v "${PWD}:/work" policy-ocr:latest --story us2 /work/test/fixtures/us2/us2_input_sample.txt --out /work/tmp_us2_out.txt
docker run --rm -v "${PWD}:/work" policy-ocr:latest --story us3 /work/test/fixtures/us3/us3_input_sample.txt --out /work/tmp_us3_out.txt
docker run --rm -v "${PWD}:/work" policy-ocr:latest --story us4 /work/test/fixtures/us4/us4_input_sample.txt --out /work/tmp_us4_out.txt
```

If `--out` is omitted, output is printed to stdout.

Windows PowerShell example:

```powershell
docker run --rm -v "${PWD}:/work" policy-ocr:latest --story us1 /work/test/fixtures/us1/us1_input_sample1.txt --out /work/tmp_us1_out.txt
```

### Docker Compose (Easier Local Runs)

This repository also includes `docker-compose.yml` so you do not need to type full `docker run` commands.

Build image through Compose:

```bash
docker compose build
```

Run with custom user-provided paths (recommended):

```bash
docker compose run --rm ocr --story us1 /work/test/fixtures/us1/us1_input_sample1.txt --out /work/tmp_us1_out.txt
docker compose run --rm ocr --story us2 /work/test/fixtures/us2/us2_input_sample.txt --out /work/tmp_us2_out.txt
docker compose run --rm ocr --story us3 /work/test/fixtures/us3/us3_input_sample.txt --out /work/tmp_us3_out.txt
docker compose run --rm ocr --story us4 /work/test/fixtures/us4/us4_input_sample.txt --out /work/tmp_us4_out.txt
```

Run shortcut services with preconfigured fixture defaults:

```bash
docker compose run --rm us1
docker compose run --rm us2
docker compose run --rm us3
docker compose run --rm us4
```

## Extending to New Stories (Handling Unknowns)

If a new mode is required (for example `us5` or another custom stage):

1. Add a strategy in `src/ocr/reporting/strategies.js`.
2. Register it in `src/ocr/reporting/reportingStageFactory.js`.
3. Add fixtures under `test/fixtures/<new-story>/`.
4. Add end-to-end and error-path tests in `test/ocr.test.js`.
5. Update this README command examples.

## Notes

- The runtime implementation uses Node built-ins only (no third-party runtime libs).
